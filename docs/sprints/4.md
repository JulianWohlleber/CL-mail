# Sprint 4 — Thread-list windowing + paging

**Lead:** David (backend), Maya (eng lead)
**Tag:** v0.1.4

## Research

**David:** "Profiled a 10,000-thread inbox in dev mode. The full list renders
in ~280 ms but every scroll re-paints all ThreadItems because the parent
re-renders on any sub-store change. We're hitting ~14 ms paint per scroll
event. Felt scrolly."

**Maya:** "Two options: virtualize, or page. Virtualization adds React-
window or hand-rolled IntersectionObserver — significant. Paging is a
60-line change. Let's page now, virtualize later if it isn't enough."

## Design

- `MAIL_LIST` IPC already accepts limit/offset → we just stop fetching
  everything at once
- Mail store: keep an `endReached` flag, fetch in chunks of 100
- MailList: an IntersectionObserver on a sentinel `<div>` at the bottom
  triggers the next chunk
- A subtle "Loading more…" footer while the next page is in-flight

## Development

- mail.store.loadMoreThreads()
- MailList: sentinel + observer + footer
- IPC layer unchanged — limit/offset already works

## Testing

- Regression test #15: store has loadMoreThreads and tracks endReached so a
  future refactor can't silently revert to "load everything"
